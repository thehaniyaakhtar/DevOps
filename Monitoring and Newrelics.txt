Logging
Logging refers to recording events, messages, and data generated by 
applications and systems. 
It is primarily used for debugging, auditing, and understanding system 
behavior over time.

Logs can include:
Error logs (failures, exceptions)
Access logs (incoming requests, endpoints hit)
Audit logs (who did what, security-related tracking)

Logs are typically written to files, external systems, or centralized 
logging platforms.

Monitoring
Monitoring involves continuously observing a system to ensure it is 
functioning correctly and efficiently. 
It focuses on real-time system health and performance.

Common metrics include:
CPU usage
Memory usage
Network traffic
Disk I/O

Monitoring answers: 
“Is the system healthy right now?”

Alerts
Alerts are triggered when predefined conditions are met in your 
monitoring or logging system.

Examples:
CPU usage exceeds a threshold
Error rate spikes
Response time degrades

Alerts are sent to destinations such as:
Email
Slack / webhook
Incident management tools
Tools and Platforms

There are both managed and self-hosted solutions:
Managed (paid):
New Relic
Datadog
Amazon CloudWatch

Self-hosted:
Prometheus (metrics collection)
Grafana (dashboards)
Loki (logging)
How Monitoring Tools Work (Example: New Relic)

With tools like New Relic:
An agent runs on each machine
It collects:
System metrics (CPU, memory)
Application metrics
Data is sent to the platform for analysis
Core Entities in Monitoring Platforms
Dashboards: Visualize metrics and trends
Alert conditions: Define what to monitor
Policies: Group related alerts
Destinations: Where alerts are sent (email, Slack, etc.)
Querying Metrics (NRQL)

New Relic uses NRQL (New Relic Query Language) to query time-series data.

Used to analyze metrics over time
Can group and filter data
Example use:
Split metrics by service
Track errors per endpoint

Concept:
FACET allows grouping (like SQL GROUP BY)
Queries can be turned into graphs
APM (Application Performance Monitoring)

APM focuses on monitoring the application itself rather than just the 
infrastructure.

It tracks:
Incoming HTTP/HTTPS requests
Response times
Error rates
Database queries

This helps identify bottlenecks inside the application.
Logging in Applications (Winston)
Winston is a logging library used in Node.js.

Uses transports to send logs to different destinations:
Console
Files
External logging services

This allows centralized logging across systems.

Percentiles (P95, P99) vs Median
When measuring performance (like response time), averages can be 
misleading due to outliers.

Median (P50): Middle value → stable indicator
P95 / P99:
P95 → 95% of users experience this response time or better
P99 → 99% of users experience this response time or better

These metrics help:
Ignore extreme outliers
Understand worst-case user experience
Important Insight on Outliers
Even if a small percentage of users (e.g., 0.01%) experience very high 
latency, they:

May not affect averages significantly
But still represent real user issues

In production systems:
Percentiles are used for monitoring
Outliers are investigated separately (not ignored completely)
End-to-End Observability Flow
Application generates logs (via Winston or similar)
Monitoring agent collects system + app metrics
Data is sent to monitoring platform
Dashboards visualize trends
Alerts trigger when thresholds are exceeded
Engineers investigate using logs + metrics
Missing Concept (Added): Observability

Modern systems combine:
Logging
Monitoring
Tracing (APM)

This combined approach is called observability, which helps understand 
what, why, and where issues occur.
